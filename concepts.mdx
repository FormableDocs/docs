Most Formable integrations use **e-signatures**. You upload a reusable **template**, create a **signature request** for one signer, embed a **signing URL**, then download the **signed envelope** when they're done.

## Templates

A **template** is a document you've uploaded (PDF or DOCX) with fields placed on it — for example signature, text, date, checkbox, or amount. Templates are reusable: upload once, then create many signature requests from the same `templateId`.

- Create a template with [Create a template](/api-reference/endpoint/create-template).
- Place fields using the **editor URL** returned on creation, or fetch a fresh one with [Create a template edit URL](/api-reference/endpoint/create-template-edit-url).

See [Manage templates](/guides/templates) for the full workflow.

## Signature requests

A **signature request** is one send of a template to one signer. It captures the `templateId`, the `signer`, the `sender`, and a `status`.

| Field | Description |
| --- | --- |
| `signer` | The person who signs the document (`email` + `name`). |
| `sender` | The person or organization sending it (`email` + `name`). |
| `status` | `Created` when made, `Completed` once signed. |
| `testMode` | When `true`, the request doesn't count toward billing. |

To let someone sign, generate a short-lived **signing URL** from the signature request and embed it in an iframe. When signing finishes, download the **signed envelope** — a temporary URL to the completed PDF.

<Info>
  An **envelope** is Formable's internal record of a document prepared for a recipient. As an API consumer you work with signature requests; the signed envelope endpoint is how you get the final PDF.
</Info>

## Signing URLs are short-lived and embeddable

Signing URLs are meant to be **embedded in an iframe** in your own product. They expire (typically after one hour), so generate them on demand right before you display them rather than storing them.

<Warning>
  Generate signing URLs from your backend and pass them to the client just-in-time. Never expose your API credentials in client-side code.
</Warning>

## Test mode

Signature requests accept a `testMode` flag. Use it while integrating so activity doesn't count toward billing.

## Events

Track signing progress two ways:

- **Push** — configure a [webhook](/webhooks/overview) for real-time events such as `document_viewed` and `document_signed`.
- **Pull** — poll [signature request events](/api-reference/endpoint/get-signature-request-events) or check the signature request `status` to reconcile state.

Prefer webhooks for reacting in near real time, and use polling to backfill anything missed.

## Redlining (optional)

Most integrations don't need this. [Redlining](/guides/redlining) is a separate workflow for negotiating a DOCX contract between two parties before signing. It uses **redline requests**, member roles, and turn-based rounds — see that guide when you're ready.
